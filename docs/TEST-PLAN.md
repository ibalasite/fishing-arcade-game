## §3 Integration Tests

Integration tests use Testcontainers to spin up real PostgreSQL 15 and Redis 7 instances. Each test file uses a `beforeAll` to apply migrations and seed data, and an `afterAll` to terminate containers. Each individual test wraps mutations in a transaction that is rolled back in `afterEach` to preserve isolation.

### §3.1 Setup Pattern

```typescript
// __tests__/integration/setup.ts
import { PostgreSqlContainer, RedisContainer } from 'testcontainers';
import { runMigrations } from '../../src/db/migrations';

let pgContainer: StartedPostgreSqlContainer;
let redisContainer: StartedRedisContainer;

beforeAll(async () => {
  pgContainer = await new PostgreSqlContainer('postgres:15').start();
  redisContainer = await new RedisContainer('redis:7').start();
  process.env.DATABASE_URL = pgContainer.getConnectionUri();
  process.env.REDIS_URL = `redis://${redisContainer.getHost()}:${redisContainer.getMappedPort(6379)}`;
  await runMigrations();
}, 60_000);

afterAll(async () => {
  await pgContainer.stop();
  await redisContainer.stop();
});

// Per-test isolation: wrap in transaction, rollback in afterEach
let trx: PoolClient;
beforeEach(async () => { trx = await db.getClient(); await trx.query('BEGIN'); });
afterEach(async () => { await trx.query('ROLLBACK'); trx.release(); });
```

## §5 Performance Tests (k6)

### §5.1 SLA Gates

All SLA gates must pass before a release is promoted to production.

| Metric | SLA Target | Failure Action |
|--------|-----------|----------------|
| p99 WebSocket shoot latency | < 100ms | Block release |
| p99 room join latency | < 500ms | Block release |
| p99 REST API response time | < 500ms | Block release |
| Error rate (all requests) | < 0.1% | Block release |
| WebSocket message loss rate | < 0.01% | Block release |

### §5.2 Load Scenarios

| Scenario | Description | Duration | VUs |
|----------|-------------|----------|-----|
| **Ramp-up** | 0 → 2000 concurrent players (500 rooms × 4) | 5 minutes | 0 → 2000 |
| **Steady-state** | Maintain 2000 players, continuous shoot + join cycle | 15 minutes | 2000 |
| **Spike** | 4000 players (1000 rooms) for 1 minute | 1 minute | 4000 |
| **Soak** | 500 players for 2 hours (overnight) | 2 hours | 500 |

### §5.3 k6 Script (TypeScript)

```typescript
// k6/fishing-load-test.ts
import { check, sleep } from 'k6';
import ws from 'k6/ws';
import http from 'k6/http';
import { Trend, Rate, Counter } from 'k6/metrics';

// Custom metrics
const shootLatency = new Trend('shoot_latency_ms', true);
const joinLatency  = new Trend('join_latency_ms', true);
const errorRate    = new Rate('error_rate');
const totalShoots  = new Counter('total_shoots');

export const options = {
  scenarios: {
    ramp_up: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '5m', target: 2000 },   // ramp up
        { duration: '15m', target: 2000 },  // steady state
        { duration: '1m', target: 4000 },   // spike
        { duration: '1m', target: 0 },      // ramp down
      ],
    },
  },
  thresholds: {
    shoot_latency_ms: ['p(99)<100'],
    join_latency_ms:  ['p(99)<500'],
    http_req_duration: ['p(99)<500'],
    error_rate: ['rate<0.001'],
  },
};

// Test fixture: pre-generated JWT tokens for load test users
const TOKENS: string[] = JSON.parse(open('./fixtures/tokens.json'));

export default function () {
  const token = TOKENS[__VU % TOKENS.length];
  const headers = { Authorization: `Bearer ${token}` };

  // 1. Join room via Colyseus WebSocket
  const joinStart = Date.now();
  const url = `wss://staging.game.example.com/?token=${token}`;

  ws.connect(url, {}, (socket) => {
    joinLatency.add(Date.now() - joinStart);

    socket.on('open', () => {
      // Send join_room message
      socket.send(JSON.stringify({ type: 'join_room', nickname: `k6_user_${__VU}` }));
    });

    socket.on('message', (data: string) => {
      const msg = JSON.parse(data);

      if (msg.type === 'state_patch' && msg.state?.roomState === 'PLAYING') {
        // 2. Send shoot messages every 200ms (5/s per player)
        const fishIds: string[] = Object.keys(msg.state?.fish ?? {});
        if (fishIds.length > 0) {
          const shootStart = Date.now();
          socket.send(JSON.stringify({
            type: 'shoot',
            bulletId: crypto.randomUUID(),
            fishId: fishIds[0],
            betAmount: 10,
            cannonMultiplier: 1,
          }));
          totalShoots.add(1);

          // Wait for shoot_result
          socket.once('message', (result: string) => {
            const res = JSON.parse(result);
            if (res.type === 'shoot_result') {
              shootLatency.add(Date.now() - shootStart);
              check(res, { 'shoot_result received': (r) => r.type === 'shoot_result' });
            } else {
              errorRate.add(1);
            }
          });
        }
        sleep(0.2); // 200ms between shots
      }
    });

    socket.on('error', () => errorRate.add(1));
    socket.setTimeout(() => socket.close(), 30_000); // each VU plays for 30s
  });

  // 3. REST health check interleaved
  const healthRes = http.get('https://staging.game.example.com/health', { headers });
  check(healthRes, { 'health 200': (r) => r.status === 200 });

  sleep(1);
}
```

## §7 Test Data Management

### §7.1 Seed Data Strategy

Integration tests use a layered seeding strategy:

1. **Schema migrations** applied once per test suite via `runMigrations()` (Testcontainers fresh DB)
2. **Static reference data** seeded in `beforeAll` (privacy_policies table, jackpot_pool singleton)
3. **Per-test dynamic data** created using factory functions; rolled back in `afterEach`

No production data is ever used in tests. All test users have email domain `@test.invalid`.

### §7.2 Factory Functions

```typescript
// __tests__/factories/index.ts

export interface UserFactoryOptions {
  gold?: number;
  diamond?: number;
  deletionStatus?: 'active' | 'pending' | 'deleted';
  nickname?: string;
}

/**
 * Creates a user + wallet row in the DB.
 * Returns userId and a valid JWT for use in API/WebSocket tests.
 */
export async function createUser(
  db: Pool,
  opts: UserFactoryOptions = {}
): Promise<{ userId: string; jwt: string; nickname: string }> {
  const userId = randomUUID();
  const nickname = opts.nickname ?? `test_user_${userId.slice(0, 6)}`;
  const email = `${userId.slice(0, 8)}@test.invalid`;
  const passwordHash = await bcrypt.hash('TestPass1!', 10);

  await db.query(
    `INSERT INTO users(id, email, email_hash, nickname, password_hash, deletion_status)
     VALUES($1, $2, $3, $4, $5, $6)`,
    [userId, encrypt(email), hmac(email, TEST_HMAC_KEY), nickname, passwordHash, opts.deletionStatus ?? 'active']
  );
  await db.query(
    `INSERT INTO user_wallets(user_id, gold, diamond) VALUES($1, $2, $3)`,
    [userId, opts.gold ?? 1000, opts.diamond ?? 0]
  );

  const jwt = signJwt({ sub: userId, role: 'player' }, TEST_JWT_PRIVATE_KEY, { expiresIn: '15m' });
  return { userId, jwt, nickname };
}

/**
 * Creates a Colyseus room in WAITING state.
 * Returns roomId for use in room-level integration tests.
 */
export async function createRoom(
  server: ColyseusTestServer
): Promise<{ roomId: string; room: GameRoom }> {
  const room = await server.createRoom('game_room', {});
  return { roomId: room.roomId, room };
}

/**
 * Seeds a wallet directly (bypass service layer for test isolation).
 */
export async function setWalletBalance(
  db: Pool, userId: string, gold: number, diamond: number = 0
): Promise<void> {
  await db.query(
    'UPDATE user_wallets SET gold=$1, diamond=$2 WHERE user_id=$3',
    [gold, diamond, userId]
  );
}

/**
 * Seeds a jackpot_pool singleton row (safe to call multiple times via ON CONFLICT).
 */
export async function seedJackpotPool(db: Pool, amount: number = 10_000): Promise<void> {
  await db.query(
    `INSERT INTO jackpot_pool(id, current_amount) VALUES(1, $1)
     ON CONFLICT (id) DO UPDATE SET current_amount=$1, updated_at=NOW()`,
    [amount]
  );
}

/**
 * Seeds the privacy_policies reference table with a current version.
 */
export async function seedPrivacyPolicy(db: Pool, version: string = '1.0.0'): Promise<void> {
  await db.query(
    `INSERT INTO privacy_policies(version, content_url, effective_at)
     VALUES($1, 'https://example.com/policy', NOW())
     ON CONFLICT (version) DO NOTHING`,
    [version]
  );
}
```

### §7.3 Cleanup Patterns

**Transaction rollback** (preferred — fastest, most isolated):
```typescript
let client: PoolClient;
beforeEach(async () => {
  client = await pool.connect();
  await client.query('BEGIN');
  // Override db module to use this transaction client within the test
  jest.spyOn(db, 'query').mockImplementation((...args) => client.query(...args));
});
afterEach(async () => {
  await client.query('ROLLBACK');
  client.release();
  jest.restoreAllMocks();
});
```

**Redis cleanup** (explicit DEL, no transactions):
```typescript
afterEach(async () => {
  await redis.del('game:jackpot:pool');
  await redis.del('email_confirm:*');  // scan + delete pattern keys
});
```

**Testcontainer lifecycle** (suite-level, not per-test):
```typescript
// Containers spin up once per suite (beforeAll) and stop after all tests (afterAll).
// Do NOT restart containers per-test — it adds 10–20s per test case.
```

### §7.4 Sensitive Data Rules in Tests

- All test emails use domain `@test.invalid` — never real email domains
- JWT signing keys for tests use distinct test keys (never production keys); stored in GitHub Actions Secrets only
- `AES_KEY` and `HMAC_SECRET_KEY` for tests are fixed 32-character strings in CI env vars (not committed to source)
- Test wallets use small amounts (gold ≤ 10,000); never real-money values
- `rtp_logs`, `jackpot_history`, `transactions` inserted during tests are rolled back; never exported
